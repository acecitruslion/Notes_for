## 1. Project Overview

**Elevator pitch (memorize this):**
"I built an async Python API Gateway that sits in front of multiple backend services. It handles routing, load balancing, rate limiting, circuit breaking, health checks, and monitoring — the things every microservices setup needs but shouldn't have to duplicate in every service. **Problem statement (general):** In a microservices setup, if clients call backend services directly:
- Every service re-implements rate limiting, retries, auth, logging, metrics
- A dead backend causes client-facing errors with no failover
- No central place to see traffic, latency, or error trends
- Load isn't balanced — one instance can get hammered while others sit idle
- A buggy or malicious client can take a backend down with no protection

**Why an API Gateway specifically:** It centralizes all of these cross-cutting concerns into a single layer in front of the backends, the same role played by real tools like Kong, NGINX, Envoy, and AWS API Gateway.

- Latency-aware load balancing (not blind round robin)

- Retry logic with exponential backoff + jitter
- Prometheus metrics (`/metrics` endpoint) + Grafana dashboards

- Fully containerized with Docker Compose (gateway + 3 backends + Redis + Prometheus + Grafana)

## 2. Core Concepts (know these cold)

**API Gateway** — a single entry point that all client requests hit first, instead of hitting backend services directly. It enforces policy (rate limits, routing, security) and gives one place for observability.

**Reverse Proxy** — a server that sits in front of one or more backend servers and forwards client requests to them, returning the backend's response as if it came from the proxy itself. Your gateway *is* a reverse proxy, plus extra intelligence (LB, rate limiting, circuit breaking) layered on top.

**Asynchronous programming (`async`/`await`)** — a single-threaded event-loop model where the program doesn't block while waiting on I/O (like a network call to a backend); instead it switches to other pending work and comes back when the I/O completes. This is why one gateway process can juggle thousands of concurrent connections without a thread per request.

**Load Balancing** — distributing incoming requests across multiple backend instances. Your implementation is **latency-aware** rather than round robin: it tracks a rolling average response time per backend and routes new requests to whichever healthy backend is currently fastest (with a little random jitter so it doesn't always pick the exact same one on ties).

**Rate Limiting** — protecting backends from being overwhelmed by any single client. You use the **token bucket algorithm**: each client has a bucket that holds tokens, refills over time at a fixed rate, and each request consumes one token. No tokens left → reject with HTTP 429. Token bucket is chosen over a fixed window because it smooths bursts while still enforcing an average rate.

**Circuit Breaker** — stops sending requests to a backend that's actively failing, instead of hammering it. Three states:
- `CLOSED` — normal, requests flow through
- `OPEN` — tripped after too many failures (5 in your code); blocks requests for a cooldown window (10s)
- `HALF_OPEN` — after the cooldown, allows one trial request; success → back to `CLOSED`, failure → back to `OPEN`

**Health Checks** — a background loop that actively pings every backend (every 5 seconds, 2-second timeout) to see if it's reachable at all, and updates the load balancer's list of "up" servers accordingly.

**Docker** — packages the gateway (and each backend) with its dependencies into a container so it runs identically anywhere. `docker-compose.yml` wires up the gateway + 3 backends + Redis + Prometheus + Grafana as one multi-container system you can start with one command.

**Load Testing (k6)** — `script.js` simulates concurrent virtual users hammering the gateway for a fixed duration and logs response times/errors, which is how your ~2000 req/sec / ~80ms latency numbers were derived.

## 4. Complete Request Workflow (step-by-step — this is the core answer)

When a request hits `handle()` in `server.py`:

1. **Identify the client** — uses the caller's IP (`request.remote`) as `client_id`.
2. **Rate limit check** — `is_allowed(client_id)` looks up a Redis hash storing `tokens` and `last refill time`. It refills tokens based on elapsed time × refill rate (capped at bucket capacity). If tokens < 1 → **HTTP 429**. Otherwise consumes 1 token and proceeds.
3. **Pick a backend** — `get_next_server()`:
   - Filters the "up" (healthy) servers down to ones whose circuit breaker currently allows requests
   - Among those, picks the one with the lowest recorded rolling-average latency (+ small jitter to break ties)
   - If nothing qualifies → **HTTP 503**
4. **Forward the request** — `forward_request()` uses a shared `aiohttp.ClientSession` (connection pooling) to call the chosen backend. On failure, retries up to 3 times with exponential backoff (base delay × 2^attempt, plus jitter).
5. **Update circuit breaker** — response status ≥ 500 → `record_failure()` (may trip breaker to OPEN); otherwise `record_success()` (resets to CLOSED).
6. **Update latency** — rolling average `(old + new) / 2` so future routing decisions reflect current backend speed.
7. **Record metrics** — increments request counter, observes latency histogram, increments error counter on failure.
8. **Return response** to the client, stripping headers like `Transfer-Encoding`/`Content-Length` since aiohttp recalculates these itself.

**In parallel:** a background asyncio task (`health_check_loop`) pings every backend's root endpoint every 5 seconds with a 2-second timeout, and rebuilds the "up servers" list from whichever respond.

**One-sentence version:** Request → rate-limit check → pick best healthy backend → forward with retries → update circuit breaker/latency → record metrics → respond.

## 5. Important Code Logic (concepts, not memorization)

- **Async request forwarding:** the handler `await`s the backend call instead of blocking; while waiting on one backend, the event loop can process other incoming requests.
- **Backend selection (latency-aware, not literal round robin):** filter by health + circuit-breaker state, then take the minimum by rolling-average latency, with light randomness to avoid pile-up when latencies are near-identical.
- **Rate limiter (token bucket):** read current tokens + last refill timestamp from Redis → compute how many tokens should have refilled since then → cap at max capacity → check if ≥ 1 → deduct and allow, or reject.
- **Error handling:** retries are only for transient/backend failures (5xx or connection errors); a 4xx is treated as the client's fault, doesn't trigger retries or trip the circuit breaker.
- **Circuit breaker transitions:** count consecutive failures → threshold reached → OPEN for a fixed cooldown → after cooldown, allow exactly one trial request (HALF_OPEN) → success closes it, failure reopens it.

## 8. Interview Q&A

**"Explain your project."** → Use the elevator pitch in Section 1, then be ready to go deeper into any single piece (rate limiter, circuit breaker, etc.) if they probe.

**"Why build this when Kong/NGINX/Envoy/AWS API Gateway already exist?"**
> "This wasn't meant to replace a production-grade gateway — it was meant to understand what they do internally by building the core mechanisms myself: load balancing, rate limiting, circuit breaking, health checks, and metrics. Building it from scratch surfaced real problems these tools solve — race conditions in distributed rate limiting, why circuit breakers need multiple states, why connection pooling matters for latency — in a way just consuming a managed gateway wouldn't have."

**"Why latency-aware routing instead of round robin?"** → Round robin ignores backend health/speed, so a slow backend still gets equal traffic. Latency-aware routing self-corrects — slower backends automatically get fewer requests, improving overall tail latency.

**"What if two requests hit the token bucket at the same time?"** → Current implementation does a read-then-write on Redis, which isn't atomic — under concurrency this could let a client slightly exceed its limit. Production fix: a Redis Lua script (atomic) or `INCR`/`EXPIRE`, or the `redis-cell` module.

**"What happens if Redis goes down?"** → Right now the rate limiter would throw and the request would fail — Redis is a single point of failure. Improvement: run Redis with Sentinel/Cluster for HA, and/or explicitly decide to fail-open (allow traffic) during an outage.

**"Is the gateway itself a single point of failure?"** → Yes, as built. In production you'd run multiple gateway replicas behind a Layer-4/7 load balancer, all sharing the same Redis so rate-limit state stays consistent — this is exactly why the rate limiter uses Redis instead of in-memory counters.

**"Why check for status ≥ 500 for the circuit breaker but not 4xx?"** → 5xx means the backend itself is unhealthy — that's what the breaker protects against. 4xx is the client's fault (bad request) and doesn't indicate backend health, so it shouldn't trip the breaker.

**"Why exponential backoff with jitter on retries?"** → Plain immediate retries can cause a thundering herd — many clients retrying at once overload a backend right as it's recovering. Backoff spaces retries out over growing intervals; jitter prevents them from clustering at the same instant.

**"Aren't health checks and circuit breakers redundant?"** → No — different timescales and purposes. Health checker is active/periodic (every 5s), answers "is it reachable at all." Circuit breaker is reactive/per-request, answers "is it actually succeeding under real traffic right now," without waiting for the next health-check cycle.

**"Why a shared `ClientSession` instead of a new one per request?"** → Opening a new TCP connection per request costs a DNS lookup + TCP handshake (+TLS if HTTPS) every time. A shared session with connection pooling reuses existing connections, which is a big part of the low ~80ms latency.

**"What's the time complexity of picking a backend?"** → O(n) where n = number of healthy backends (filter, then find min latency) — trivial at the scale this project runs at (tens of backends, not thousands).

**"Why HALF_OPEN instead of jumping straight from OPEN to CLOSED?"** → Going straight to CLOSED risks flooding a still-recovering backend with full traffic again. HALF_OPEN sends one trial request first — success closes fully, failure reopens.

**"Why rate-limit by IP? What's the downside?"** → Simple to implement, but multiple users behind the same NAT/corporate proxy share an IP (over-restrictive), and an attacker can rotate IPs to dodge limits (under-restrictive). Real systems often key by API key or authenticated user ID instead or in addition.

**"How would you test this?"** → Unit tests per component (circuit breaker state transitions, token bucket math), integration tests (spin up backends + gateway, hit real endpoints), and load testing (already have the k6 script).

**"How would you scale this to millions of requests/day or a high req/sec burst?"**
1. Horizontal scaling: run N stateless gateway replicas behind a load balancer (shared state already lives in Redis, not in-process, so this is straightforward)
2. Move Redis to Cluster/Sentinel so it's not a bottleneck or SPOF
3. Scale/auto-scale backend instances based on CPU/latency
4. Consider a local + Redis hybrid for rate-limit checks to cut round-trips if strict global accuracy isn't critical
5. Offload non-critical work (logging/analytics) to a message queue so the hot path stays fast
6. Add caching for read-heavy idempotent endpoints
7. Move from Docker Compose to Kubernetes for auto-scaling, self-healing, rolling deploys
8. Scale observability (Prometheus federation or a managed backend) since a single Prometheus struggles at high volume
9. Tune connection limits, OS file-descriptor limits, Redis `maxclients`
10. For global scale, deploy gateway + backends multi-region with geo-DNS routing

**"Explain CAP theorem in context of this project."** → Redis (rate-limit state) is a shared store; under a network partition you must choose consistency (reject requests when uncertain) or availability (allow requests, risk slightly exceeding the true limit). The current implementation implicitly favors availability/simplicity over strict consistency.

## Terminology Cheat Sheet (quick definitions)

| Term | Definition |
|---|---|
| Reverse Proxy | Sits in front of backends, forwards client requests to them |
| Load Balancing | Spreading requests across multiple servers |
| Latency-aware routing | Sends traffic to whichever healthy backend is currently fastest |
| Token Bucket | Rate-limiting algorithm: tokens refill over time, each request consumes one |
| Circuit Breaker | Temporarily stops traffic to a failing service (CLOSED/OPEN/HALF_OPEN) |
| Exponential Backoff + Jitter | Retry delays grow (and randomize) to avoid overwhelming a recovering service |
| Health Check | Periodic liveness probe |
| Connection Pooling | Reusing TCP connections instead of opening new ones per request |
| Async I/O / Event Loop | Single-threaded model that doesn't block on I/O, switches to other work while waiting |
| Distributed Rate Limiting | Rate limiting that works correctly across multiple gateway replicas via shared (Redis) state |
| RED Method | Monitoring convention: track Rate, Errors, Duration |
| Idempotency | An operation that gives the same result if repeated (relevant to whether it's safe to retry it) |
| SPOF | Single Point of Failure |
| Observability | Understanding a system's internal state from its metrics/logs/traces |
